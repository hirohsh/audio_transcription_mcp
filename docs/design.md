# 音声文字起こし MCP サーバー 設計書

## 1. 概要

本システムは、OpenAI Whisper モデル (ONNX 形式) を使用して音声ファイルの文字起こしを行う MCP (Model Context Protocol) サーバーです。Rust で実装され、stdio トランスポートを介して MCP クライアント (Claude Desktop 等) と通信します。

## 3. コンポーネント詳細

### 3.1 設定管理 (config.rs)

環境変数から設定を読み込み、バリデーションを行います。

| 環境変数 | 説明 | デフォルト値 | 必須 |
|----------|------|-------------|------|
| `MCP_WORK_DIR` | 作業ディレクトリ制限 | カレントディレクトリ | No |
| `MCP_MAX_FILE_SIZE_MB` | 最大ファイルサイズ (MB) | 25 | No |
| `MCP_MODEL_DIR` | ONNX モデルディレクトリ | `./models` | No |
| `MCP_LOG_LEVEL` | ログレベル | `info` | No |

**パス検証フロー:**
1. 入力パスを正規化 (canonicalize)
2. `MCP_WORK_DIR` のプレフィックスを確認
3. パストラバーサル攻撃を防止

### 3.2 音声デコード (audio.rs)

`symphonia` クレートを使用して多形式の音声ファイルをデコードします。

**対応形式:**
- WAV (PCM)
- MP3
- FLAC
- OGG (Vorbis)

**処理フロー:**
1. ファイルを開きフォーマットをプローブ
2. デフォルトトラックを選択
3. パケットを順次デコード
4. マルチチャンネルの場合はモノラルに平均化
5. `rubato` クレートで 16kHz にリサンプリング

### 3.3 メルスペクトログラム (mel.rs)

Whisper モデルの学習時と同一のパラメータでメルスペクトログラムを計算します。

**パラメータ:**
- サンプルレート: 16,000 Hz
- FFT サイズ: 400
- ホップ長: 160 (10ms)
- 窓長: 400 (25ms)
- メルバンド数: 80
- チャンク長: 30 秒 (N_FRAMES = 3000)

**処理:**
1. Hann 窓を適用
2. FFT で周波数スペクトルを計算
3. メルフィルターバンクを適用
4. 対数スケールに変換
5. 正規化

### 3.4 トークナイザー (tokenizer.rs)

HuggingFace `tokenizers` クレートを使用して Whisper のトークナイザーを処理します。

**特殊トークン:**
- `<|startoftranscript|>` (50258): 文字起こし開始
- `<|endoftext|>` (50257): 文字起こし終了
- `<|transcribe|>` (50359): 文字起こしタスク
- `<|notimestamps|>` (50363): タイムスタンプなし
- 言語トークン (50259-50357): 99 言語対応

**プロンプト構成:**
```
<|startoftranscript|> [<|language|>] <|transcribe|> <|notimestamps|>
```

### 3.5 Whisper 推論エンジン (whisper.rs)

ONNX Runtime を使用してエンコーダー・デコーダーモデルを実行します。

**推論フロー:**
1. メルスペクトログラムを 30 秒チャンクに分割・パディング
2. エンコーダーでオーディオ特徴量を抽出
3. デコーダーで自己回帰的にトークンを生成 (最大 448 ステップ)
4. Greedy デコーディング (各ステップで最大確率トークンを選択)
5. EOT トークンで生成を終了
6. トークン列をテキストに変換

### 3.6 GPU 実行戦略

ONNX Runtime の Execution Provider を使用して GPU を優先的に利用します。

| OS | GPU Provider | フォールバック |
|----|-------------|---------------|
| macOS | CoreML | CPU |
| Windows | DirectML | CPU |
| Linux | CUDA | CPU |

**初期化フロー:**
1. OS を検出
2. 対応する GPU Execution Provider で初期化を試行
3. 失敗した場合は CPU にフォールバック
4. 使用中の Provider をログに出力

### 3.7 MCP ツール定義 (tools.rs)

`rmcp` クレートを使用して MCP ツールを定義します。

**ツール: `transcribe_audio`**

入力スキーマ:
```json
{
  "type": "object",
  "properties": {
    "file_path": {
      "type": "string",
      "description": "Path to the audio file (must be within the allowed work directory)"
    },
    "language": {
      "type": "string",
      "description": "Language code (e.g., 'ja', 'en'). If omitted, language is auto-detected."
    }
  },
  "required": ["file_path"]
}
```

## 4. データフロー

```
音声ファイル (wav/mp3/flac/ogg)
    │
    ▼
パス検証 (config.rs)
    │ ・作業ディレクトリ内チェック
    │ ・ファイルサイズチェック
    ▼
音声デコード (audio.rs)
    │ ・フォーマット検出
    │ ・PCM デコード
    │ ・モノラル変換
    │ ・16kHz リサンプリング
    ▼
メルスペクトログラム (mel.rs)
    │ ・STFT 計算
    │ ・メルフィルター適用
    │ ・対数スケール変換
    │ ・30秒チャンク分割
    ▼
エンコーダー (whisper.rs)
    │ ・ONNX Runtime 推論
    │ ・オーディオ特徴量抽出
    ▼
デコーダー (whisper.rs)
    │ ・自己回帰生成
    │ ・Greedy デコーディング
    │ ・EOT で終了
    ▼
トークン→テキスト (tokenizer.rs)
    │
    ▼
文字起こし結果 (テキスト)
```

## 5. セキュリティ考慮事項

1. **パストラバーサル防止**: `canonicalize()` でパスを正規化し、`MCP_WORK_DIR` のプレフィックスを検証
2. **ファイルサイズ制限**: メモリ枯渇攻撃を防止 (デフォルト 25MB)
3. **入力バリデーション**: 対応フォーマット以外のファイルは `symphonia` レベルで拒否
4. **ログ出力先**: stderr に出力し、stdio の MCP 通信を汚染しない

## 6. エラーハンドリング

| エラー種別 | 対応 |
|-----------|------|
| ファイル未存在 | 「File not found」エラーメッセージ |
| アクセス拒否 (パス外) | 「Access denied」エラーメッセージ |
| サイズ超過 | サイズ情報を含むエラーメッセージ |
| 非対応フォーマット | 「Unsupported audio format」エラーメッセージ |
| モデル未配置 | 起動時にエラーで終了 |
| GPU 初期化失敗 | CPU にフォールバック (ログで通知) |
